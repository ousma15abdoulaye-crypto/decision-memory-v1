# ADR-M12-001 — M12 Procurement Document & Process Recognizer Architecture

**Status:** Accepted  
**Date:** 2026-03-30  
**Decision Makers:** CTO (AO)  

## Context

The existing Pass 1 (router) is a simple deterministic classifier with 7 regex rules and LLM fallback, producing TaxonomyCore (12 values) + DocumentRole (8 values). This is insufficient for procurement document analysis at scale.

M12 introduces a 10-layer cognitive engine (7 active + 3 handoffs) to recognize procurement documents, assess their validity, compute conformity signals, and link them within procurement processes.

## Decision

### 1. Decompose Pass 1 Into Sub-Passes

Pass 1 is decomposed into 4 sub-passes (1A/1B/1C/1D) integrated into the existing orchestrator FSM. The old `pass_1_router.py` is NOT deleted — it is deprecated.

### 2. Config-Driven Architecture

All detection rules live in YAML under `config/`. Adding a new document type or framework = adding a YAML file, zero code change.

### 3. TracedField Generic

A new `TracedField[T]` Pydantic generic enforces `value + confidence + evidence` for all critical outputs. M12 internal confidence is [0.0, 1.0]; discretized to {0.6, 0.8, 1.0} at export boundary.

### 4. Read-Only Couche B Connection

M12 reads from Couche B assets (dictionary, mercuriales) in read-only mode. No writes to Couche B tables.

### 5. New `src/procurement/` Module

All M12 engines live in `src/procurement/`, separate from `src/annotation/`. Pass functions in `src/annotation/passes/` bridge the two.

## Consequences

- 18 new Python files in `src/procurement/`
- 4 new pass files in `src/annotation/passes/`
- 20 YAML config files in `config/`
- 16 test files
- 1 Alembic migration (054)
- Orchestrator FSM gains 4 new states (PASS_1A_DONE through PASS_1D_DONE)
- `rapidfuzz` and `pyyaml` added to requirements.txt
